# <a name="markdown-style-guide-for-powershell-docs"></a>Руководство по стилю Markdown для документации по PowerShell

В этой статье приводятся некоторые указания по стилю, относящиеся к содержимому документации по PowerShell.

Другие рекомендации по стилю написания см. в [руководстве по стилю Майкрософт](https://docs.microsoft.com/style-guide/welcome/).

## <a name="metadata"></a>Метаданные

Этот шаблон содержит примеры синтаксиса Markdown, а также инструкции по указанию метаданных.
При создании файла Markdown скопируйте приведенный шаблон в новый файл, укажите метаданные, как описано ниже, и установите заголовок H1 над названием статьи.

Необходимый блок метаданных приводится в следующем примере блока метаданных:

```md
---
author: [GITHUB USERNAME]
description: [usually a summary of your first paragraph. It gets displayed in search results, and can help drive the correct traffic if well written.]
ms.date: [CREATION/UPDATE DATE - mm/dd/yyyy]
title: [ARTICLE TITLE]
---
# The H1 should not be the same as the title, but should describe the article contents
```

- Между двоеточием (:) и значением в элементе метаданных **должен** быть пробел.
- Двоеточия в значении (например, заголовке) прерывают анализ метаданных. В этом случае заключите заголовок в двойные кавычки (например, `title: "Writing .NET Core console apps: An advanced step-by-step guide"`).
- **author**: В этом поле необходимо указать **имя пользователя автора на GitHub**.
- **description**: Описание содержимого статьи. Обычно описание отображается на странице результатов поиска, но не используется для упорядочивания результатов. Длина: 115–145 символов с пробелами.
- **ms.date**: Дата последнего значительного изменения. Измените это значение в существующей статье, если вы пересмотрели и обновили всю статью. Небольшие исправления, вроде опечаток, не считаются обновлением.
- **title**: Отображается в результатах поиска. Название не должно совпадать со значением в заголовке H1 и превышать 60 символов в длину.

К каждой статье прикрепляются другие метаданные, но обычно мы применяем значения большинства метаданных на уровне папки в файле **docfx.json**.

## <a name="product-terminology"></a>Названия продуктов

Существует несколько вариантов PowerShell. В этой таблице поясняется ряд терминов, используемых в отношении PowerShell.

- **PowerShell** — термин по умолчанию. Это название предоставляемого нами продукта. Термин PowerShell можно использовать в отношении любого выпуска продукта.
- **PowerShell Core** — это продукт PowerShell на основе среды выполнения .NET Core (CoreCLR) для любой из поддерживаемых платформ.
- **Windows PowerShell** — это продукт PowerShell на основе среды выполнения (CLR) .NET Framework. Windows PowerShell поставляется только с Windows и требует наличия полнофункциональной среды выполнения .NET Framework.

В общем случае термин "Windows PowerShell" в документации можно сокращать до "PowerShell".
Термин "Windows PowerShell" **не следует** сокращать при обсуждении продукта, предназначенного для Windows.

## <a name="basic-markdown-gfm-and-special-characters"></a>Базовый Markdown, GFM и специальные символы

О базовых принципах Markdown, GitHub Flavored Markdown (GFM) и конкретных расширениях OPS можно прочитать в статье [Справочник по Markdown](../markdown-reference.md).

Markdown использует для форматирования специальные символы, например \*, \` и \#. Если вы хотите включить такой символ в свою статью, сделайте одно из двух:

- Поставьте обратную косую черту перед символом, чтобы экранировать его (например, `\*` для \*)
- Используйте [код сущности HTML](http://www.ascii.cl/htmlcodes.htm) для символа (например, `&#42;` для &#42;).
- В файлах Markdown должен использоваться обычный текст ASCII или кодировка UTF-8. Однако старайтесь не использовать многобайтовые символы UTF-8. Вместо этого используйте сущность HTML. Например, для символа авторского права используйте `&copy;`.
  Он отображается как "&copy;".

## <a name="hyperlinks"></a>Гиперссылки

- Старайтесь не использовать чистые URL-адреса. В ссылках следует использовать синтаксис разметки Markdown `[friendlyname](url-or-path)`.
- Ссылки должны иметь понятные имена. Обычно это заголовки соответствующих статей.
- Все элементы в разделе "Связанные ссылки" внизу статьи должны быть снабжены гиперссылками.
- Используйте относительные ссылки на другое содержимое, размещенное на сайте **docs.microsoft.com**.
- Не используйте обратные галочки (грависы), полужирное начертание или другие элементы разметки в квадратных скобках гиперссылки.

Более подробные сведения о ссылках см. в разделе [Использование ссылок в документации](../how-to-write-links.md).

## <a name="filenames"></a>Имена файлов

Имена файлов подчиняются указанным ниже правилам.

- Содержат только буквы, цифры и дефисы.
- Должны отсутствовать пробелы и знаки препинания. Для разделения слов и чисел в имени файла следует использовать дефисы.
- Следует использовать конкретные команды действий, такие как develop, buy, build, troubleshoot. Использование слов, оканчивающихся на -ing, не допускается.
- Нельзя использовать служебные (короткие) слова, такие как a, and, the, in, or и т. д.
- Имя должно быть основано на разметке Markdown и иметь расширение файла .md.
- Имя файла должно быть коротким в разумных пределах. Оно будет частью URL-адреса статьи.

## <a name="blank-lines-spaces-and-tabs"></a>Пустые строки, пробелы и символы табуляции

Удаляйте дублирующиеся пустые строки. Несколько пустых строк отображаются в HTML как одна, поэтому использовать их нет смысла.

Кроме того, в Markdown пустые строки означают конец блока. Между блоками Markdown разных типов (например, абзацем и списком или заголовком) должна быть одна пустая строка.

> [!NOTE]
> Пробелы в Markdown имеют значение. Всегда используйте пробелы вместо жесткой табуляции. Удаляйте лишние пробелы в конце строк.

## <a name="titles-and-headings"></a>Названия и заголовки

Используйте только [заголовки в стиле ATX][atx] (то есть выделенные символами #, а не `=` или `-`).

- Используйте регистр как в предложении: с прописной буквы следует начинать только имена собственные, а также первые слова в названиях или заголовках.
- Между символом `#` и первым словом заголовка должен быть один пробел.
- Перед заголовком и после него должна быть одна пустая строка.
- В каждом документе должен быть только один заголовок H1.
- Уровни заголовков должны увеличиваться последовательно — не пропускайте уровни.
- Не используйте обратные галочки, полужирное начертание или другие элементы разметки в заголовках.

> [!CAUTION]
> Схема, применяемая в [PlatyPS][platyPS] для справки по командлетам, требует наличия определенных заголовков H2 и H3. Добавление или удаление заголовков может привести к ошибкам при сборке. Дополнительные сведения см. в [руководстве по стилю для справки по командлетам](powershell-style-reference.md).

## <a name="limit-line-length-to-100-characters"></a>Ограничьте длину строк 100 символами

Это упростит вывод изменений и истории в командной строке.

Данное правило не соблюдалось для большей части существующего содержимого в документации по PowerShell, однако мы работаем над исправлением этой ситуации. Вы можете помочь нам в этом. [Расширение расплавления][reflow] в Visual Studio Code (VSCode) позволяет легко переформатировать абзацы так, чтобы это ограничение соблюдалось.

## <a name="lists"></a>Списки

Если список должен содержать несколько предложений или абзацев, можно использовать вместо него подзаголовки.

Перед списком и после него должна быть одна пустая строка.

### <a name="unordered-lists"></a>Неупорядоченные списки

Ставить точки в конце элементов списка следует, только если они содержат несколько предложений. В качестве маркеров для элементов списка используйте символы дефиса (`-`). Это позволяет избежать путаницы с разметкой полужирного шрифта или курсивного начертания, в которой используется звездочка (`*`). Чтобы включить абзац или другие элементы в элемент маркированного списка, добавьте разрыв строки и выровняйте отступы с первым символом после маркера.

Например:

```markdown
This is a list that contain sub-elements under a bullet item.

- First bullet item

  Sentence explaining the first bullet.

  - Sub-bullet item

    Sentence explaining the sub-bullet.

- Second bullet item
- Third bullet item
```

Это маркированный список, элемент которого имеет вложенные элементы.

- Первый элемент маркированного списка

  Предложение, поясняющее первый элемент.

  - Вложенный элемент маркированного списка

    Предложение, поясняющее вложенный элемент.

- Второй элемент маркированного списка
- Третий элемент маркированного списка

### <a name="ordered-lists"></a>Упорядоченные списки

Чтобы включить абзац или другие элементы в элемент нумерованного списка, выровняйте отступы с первым символом после номера элемента.

```markdown
1. For the first element, insert a single space after the 1. Long sentences should be
   wrapped to the next line and must line up with the first character after the numbered
   list marker.

   To include a second element (like this one), insert a line break after the first
   and align indentations. The indentation of the second element must line up with
   the first character after the numbered list marker. For single digit items, like
   this one, you indent to column 4. For double digits items, for example item number
   10, you indent to column 5.

1. The next numbered item starts here.
```

Например, нужно получить такой результат:

> 1. Для первого элемента добавьте один пробел после цифры 1. Длинные предложения должны переноситься по строкам, причем все строки должны быть выровнены по первому символу после номера элемента списка.
>
>    Чтобы включить еще один элемент (например, как этот), добавьте разрыв строки после первого и выровняйте отступы. Отступ второго элемента должен быть выровнен по первому символу после номера элемента списка. Для элементов с однозначными номерами, таких как этот, необходимо отступить до столбца 4. Для элементов с двузначными номерами, например 10, необходимо отступить до столбца 5.
>
> 1. Следующий нумерованный элемент начинается здесь.

Для всех элементов в нумерованном списке должен использоваться номер `1.` вместо увеличения номера для каждого элемента.
При преобразовании разметки Markdown для просмотра значение увеличивается автоматически. Это упрощает переупорядочение элементов и позволяет стандартизировать отступы вложенного содержимого.

## <a name="formatting-command-syntax-elements"></a>Форматирование элементов синтаксиса команд

- В именах командлетов PowerShell используется [регистр Pascal][pascal-case]. Глаголы от существительных отделяются дефисами. Всегда используйте полные имена командлетов и параметров в регистре Pascal. Используйте псевдонимы, только если речь идет именно о них.

- В абзаце ключевые слова языка, имена командлетов и ссылки на переменные должны заключаться в грависы (`` ` ``). Имена свойств, параметров и классов должны выделяться **полужирным шрифтом**.

  Например:

  ~~~markdown
  The following code assigns the output of `Get-ChildItem` to the `$files` variable.

  ```powershell
  $files = Get-ChildItem C:\Windows
  ```
  ~~~

- Если параметр указывается по имени, он должен быть выделен **полужирным шрифтом**. Если иллюстрируется использование параметра с префиксом в виде дефиса, параметр должен быть заключен в грависы. Например:

  ```markdown
  The parameter's name is **Name**, but it is typed as `-Name` when used on the command
  line as a parameter.
  ```

- При упоминании внешних команд (исполняемых файлов, скриптов и т. д.) имя команды должно быть полужирным, состоять из прописных букв (в начале предложения первая буква прописная) и включать в себя соответствующее расширение имени файла. Например:

  ```markdown
  For example, on Windows systems, you can use the `net start` and `net stop` commands
  to start and stop a service. **Sc.exe** is another service control tool for Windows.
  That name does not fit into the naming pattern for the **net.exe** service commands.
  ```

- Если приводится пример использования внешней команды, он должен быть заключен в грависы.
  При совпадении имени с псевдонимом в пример команды необходимо включить расширение имени файла. Например:

  ```markdown
  To start the spooler service on a remote computer named DC01, you type `sc.exe \\DC01 start spooler`.
  ```

- При написании концептуальной статьи (а не справочного содержимого) первое упоминание имени командлета должно быть гиперссылкой на документацию по командлету. Не используйте обратные галочки (грависы), полужирное начертание или другие элементы разметки в квадратных скобках гиперссылки.

  Например:

  ```markdown
  This [Write-Host](/powershell/module/Microsoft.PowerShell.Utility/Write-Host) cmdlet
  uses the **Object** parameter to ...
  ```

  Дополнительные сведения см. в разделе [Гиперссылки](#hyperlinks) руководства по стилю.

## <a name="images"></a>Изображения

Для включения изображения используется следующий синтаксис:

```Syntax
![[alt text]](<folderPath>)
```

Пример.

```markdown
![Introduction](./media/overview/Introduction.png)
```

Где `alt text` — краткое описание изображения, а `<folder path>` — относительный путь к нему. Заменяющий текст необходим для средств чтения с экрана для слабовидящих. Он также удобен при возникновении ошибок на сайте, из-за которых не удается отобразить изображение.

Изображения должны храниться в папке `media/<article-name>`, вложенной в папку со статьей. Одно и то же изображение не следует использовать в нескольких статьях. В папке `media` создайте вложенную папку, имя которой совпадает с именем файла статьи. Скопируйте изображения для статьи в эту новую папку. Если изображение используется в нескольких статьях, его копия должна быть в папке с изображениями каждой из них. Это нужно для того, чтобы при изменении изображения в одной статье в остальных оно не менялось.

Некоторые изображения, например логотипы и значки, должны использоваться в нескольких статьях. Такие изображения хранятся в папке `/media/shared` в корне репозитория.

Для изображений поддерживаются следующие типы файлов: PNG, GIF, JPEG, JPG, SVG.
